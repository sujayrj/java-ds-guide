Big O Notation & Complexity Analysis

Before picking a data structure, you need a language to describe performance. Big O is that language — learn it once, use it forever.

What Is Big O Notation?

Big O notation describes how the running time or memory usage of an algorithm scales as the input size (n) grows toward infinity. It is not the exact number of operations — it is the growth rate.

Think of it like this: if you double the input, does the work double? Quadruple? Stay the same? That relationship is what Big O captures.

Imagine a bank with 1,000,000 customers. If you need to find one customer:
• Scanning every record one-by-one = O(n) — 1M checks in the worst case
• Using a sorted index and binary search = O(log n) — only ~20 checks
• Looking up by account ID in a hash table = O(1) — found immediately

The input size is 1M customers. Big O tells you how the cost grows if you had 2M, 10M, or 100M.

Rules for Calculating Big O

Rule 1 — Drop Constants

O(2n) → O(n). We don't care that you loop twice; we care that it's linear.

Rule 2 — Drop Non-Dominant Terms

O(n² + n) → O(n²). As n grows, the n² dominates.

Rule 3 — Sequential Steps Add

void processPayment(List<Payment> payments) {
    // Step 1: O(n)
    for (Payment p : payments) validate(p);

    // Step 2: O(n)
    for (Payment p : payments) settle(p);

    // Total: O(n) + O(n) = O(2n) → O(n)
}

Rule 4 — Nested Steps Multiply

void findDuplicates(List<String> accounts) {
    for (String a : accounts) {          // O(n)
        for (String b : accounts) {      // O(n) inside
            if (a.equals(b)) log(a, b);  // O(n * n) = O(n²)
        }
    }
}

Best, Average, and Worst Case

Big O is typically about the worst case — the maximum work the algorithm could ever do. But you'll encounter three distinct cases:

CaseNotationMeaningExample (Linear Search)
BestΩ (Omega)Minimum work possibleElement is first — Ω(1)
AverageΘ (Theta)Typical/expected workElement is in the middle — Θ(n/2) → Θ(n)
WorstO (Big O)Maximum work possibleElement not found — O(n)

Space Complexity

Space complexity measures how much additional memory an algorithm uses as input grows. It does not include the input itself (that's input space) — it measures auxiliary space.

// O(1) space — no extra memory allocated
int sum(int[] arr) {
    int total = 0;
    for (int x : arr) total += x;
    return total;
}

// O(n) space — creates a new list of size n
List<Integer> doubled(int[] arr) {
    List<Integer> result = new ArrayList<>();
    for (int x : arr) result.add(x * 2);
    return result;
}
💡
Time–Space Tradeoff

In real systems you often trade one for the other. A HashMap uses O(n) space to give you O(1) time on lookups. Caching (like Redis) is this tradeoff at infrastructure scale — spend memory to save time.

Amortised Complexity

Some operations are occasionally expensive but cheap on average. The classic example is ArrayList.add(). Normally adding an element is O(1), but occasionally the backing array is full and the list must resize — copying all elements to a new, larger array — which is O(n). However, since this happens rarely (roughly every n adds), the amortised cost across all adds is still O(1).

📌
Amortised = "averaged over many operations"

Think of it like a bank account: most days you spend little, but occasionally you pay a big bill. The average daily spend is still low. Same idea — ArrayList.add() is amortised O(1).

Chapter 0 Quiz

5 Questions

Q1. You have an algorithm that runs in O(n² + 3n + 42). What is the simplified Big O?

Reveal Answer
O(n²). We drop constants (42, 3) and non-dominant terms (3n), leaving only the fastest-growing term: n².

Q2. An operation is O(1) time but O(n) space. Is this always better than O(n) time O(1) space?

Reveal Answer
Not always. It depends on whether time or memory is the bottleneck. In memory-constrained environments (embedded systems, certain cloud pricing models), O(n) space can be worse. Big O alone doesn't decide — context matters.

Q3. If binary search is O(log n), how many checks does it take to find something in 1 billion records?

Reveal Answer
log₂(1,000,000,000) ≈ 30. Only 30 comparisons. This is why search engines and databases use sorted indexes — logarithmic search is extraordinarily efficient at scale.

Q4. What's the difference between O(n) and O(n log n)?

Reveal Answer
O(n) grows linearly. O(n log n) grows slightly faster — for 1M elements, O(n) = 1M ops and O(n log n) ≈ 20M ops. O(n log n) is the typical cost of efficient sorting algorithms like merge sort and Tim sort (used in Collections.sort).

Q5. Why is ArrayList.add() said to be "amortised O(1)" even though resizing costs O(n)?

Reveal Answer
Because resizing happens infrequently — roughly once every n adds. When you spread that O(n) cost across all n adds that triggered the resize, each add costs roughly O(1) on average. Amortised analysis averages cost across a sequence of operations.
  • Big O describes growth rate of time/space with respect to input size — not exact counts
  • Dominant growth classes: O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ)
  • Sequential steps add; nested steps multiply
  • Worst-case Big O is the most important for production safety guarantees
  • Space complexity measures extra memory used — not input size
  • Amortised cost = average cost over a sequence of operations (e.g. ArrayList.add)

Arrays — The Foundation of Everything

Every other data structure is built on top of arrays. Understanding arrays at a deep level is non-negotiable.

An array is a row of numbered lockers in a bank vault. Each locker has a fixed number, starting at 0. You can jump directly to locker #847 in one step — you don't walk past 846 lockers. However, the vault was built with a fixed number of lockers. If you need more, you have to build an entirely new vault and move everything over.

Internal Structure

An array is a contiguous block of memory. When you declare int[] arr = new int[5], the JVM allocates 5 consecutive memory addresses (4 bytes each for int = 20 bytes total). Element access works by simple arithmetic: address_of_element = base_address + (index × element_size).

This is why array access is always O(1) — no matter the index, one calculation gives you the memory address directly.

// ── Declaration and Initialisation ──
int[] amounts = new int[5];              // [0, 0, 0, 0, 0]
int[] balances = {1000, 2500, 750};   // inline init
String[] accounts = new String[3];       // [null, null, null]

// ── Access: O(1) ──
int first = balances[0];               // 1000
int last  = balances[balances.length - 1]; // 750

// ── Modification: O(1) ──
balances[1] = 3000;                    // direct assignment

// ── Traversal: O(n) ──
for (int i = 0; i < balances.length; i++) {
    System.out.println("Balance at index " + i + ": " + balances[i]);
}

// ── 2D Arrays (Matrix) ── common in banking for FX matrices ──
double[][] fxRates = new double[5][5]; // 5x5 currency pair matrix
fxRates[0][1] = 1.08;               // USD → EUR
fxRates[0][2] = 155.2;              // USD → JPY

Key Arrays API

import java.util.Arrays;

int[] payments = {500, 200, 800, 100, 650};

// Sort — O(n log n) using Dual-Pivot QuickSort
Arrays.sort(payments);
// → [100, 200, 500, 650, 800]

// Binary search — O(log n) — ARRAY MUST BE SORTED FIRST
int idx = Arrays.binarySearch(payments, 500); // returns index 2

// Copy — O(n)
int[] copy = Arrays.copyOf(payments, payments.length);
int[] partial = Arrays.copyOfRange(payments, 1, 4); // [200, 500, 650]

// Fill — O(n)
Arrays.fill(payments, 0); // all elements set to 0

// Compare — O(n)
boolean same = Arrays.equals(copy, payments);

// Stream — bridge to functional operations
int total = Arrays.stream(payments).sum(); // O(n)
int max   = Arrays.stream(payments).max().getAsInt();

// Convert to List — O(n)
List<Integer> list = Arrays.asList(1, 2, 3); // fixed-size!
⚠️
Arrays.asList() returns a FIXED-SIZE list

You can update elements but you cannot add or remove elements — it will throw UnsupportedOperationException. To get a mutable list: new ArrayList<>(Arrays.asList(...)).

When to Use Arrays

✅ Use Arrays When...
  • Size is known and fixed at compile/runtime
  • You need maximum performance (no boxing overhead)
  • Working with primitives (int[], double[])
  • Implementing algorithms (sorting, DP tables)
  • Representing matrices or grids (2D arrays)
❌ Avoid Arrays When...
  • Size changes dynamically — use ArrayList
  • Frequent insertions/deletions in the middle
  • You need rich collection API methods
  • Working with generics (type safety complications)

Production Example — Payment Amount Bucketing

/**
 * Bank risk system: bucket transactions into value ranges.
 * Uses arrays for speed — fixed-size buckets known in advance.
 */
@Service
public class TransactionBucketService {

    private static final int[] THRESHOLDS = {1_000, 10_000, 100_000, Integer.MAX_VALUE};
    private static final String[] BUCKET_NAMES = {"Low", "Medium", "High", "Whale"};

    public String classify(int amount) {
        for (int i = 0; i < THRESHOLDS.length; i++) {
            if (amount <= THRESHOLDS[i]) {
                return BUCKET_NAMES[i];
            }
        }
        return "Unknown";
    }

    public int[] countByBucket(List<Transaction> txns) {
        int[] counts = new int[THRESHOLDS.length]; // O(1) space
        for (Transaction t : txns) {              // O(n) time
            for (int i = 0; i < THRESHOLDS.length; i++) {
                if (t.getAmount() <= THRESHOLDS[i]) {
                    counts[i]++;
                    break;
                }
            }
        }
        return counts; // O(n) total
    }
}

ArrayList — Dynamic Arrays Done Right

ArrayList is your default workhorse. Know its internals, performance characteristics, and edge cases cold.

ArrayList is like a bank safe with expandable capacity. It starts with room for 10 items. When you add an 11th, it silently builds a new safe that's 50% larger, moves everything over, and continues. From the outside you just keep adding. The rare expensive move (resize) is hidden and infrequent — giving you effectively O(1) amortised adds.

Internal Mechanics

ArrayList wraps a plain Object[] array internally. It tracks two things: the backing array and size (the count of actual elements). When size == array.length, it resizes to array.length * 1.5. This growth strategy minimises total copies over many adds.

// Simplified internal representation (conceptual)
class ArrayList<E> {
    Object[] elementData; // backing array, default size 10
    int size = 0;         // actual number of elements

    void add(E e) {
        if (size == elementData.length) grow(); // expensive but rare
        elementData[size++] = e;               // O(1) normally
    }

    void grow() {
        int newCapacity = elementData.length + (elementData.length >> 1); // × 1.5
        elementData = Arrays.copyOf(elementData, newCapacity);            // O(n)
    }
}

Critical Pitfalls

🚫
ConcurrentModificationException — The most common ArrayList trap

Never modify an ArrayList while iterating it with a for-each loop. Use Iterator.remove() or removeIf() instead.

// ❌ BAD — throws ConcurrentModificationException
for (String acc : accounts) {
    if (acc.startsWith("INACTIVE")) accounts.remove(acc);
}

// ✅ GOOD — Using Iterator
Iterator<String> it = accounts.iterator();
while (it.hasNext()) {
    if (it.next().startsWith("INACTIVE")) it.remove();
}

// ✅ BEST — Using removeIf (Java 8+)
accounts.removeIf(acc -> acc.startsWith("INACTIVE"));

LinkedList — The Chain of Nodes

LinkedList excels at head/tail insertions and deletions. Its random access performance is its biggest weakness.

LinkedList is like a chain of SWIFT messages where each message contains a pointer to the next message in the chain (and the previous one). To find message #500, you must start at message #1 and follow 499 links. But to insert or remove a message anywhere in the chain, you just re-wire two links — O(1) once you're at the right position.

Internal Structure — Doubly Linked List

Java's LinkedList is a doubly-linked list. Each node holds a value, a reference to the next node, and a reference to the previous node. The list tracks both the head (first node) and tail (last node) directly.

// Internal node (simplified)
static class Node<E> {
    E item;
    Node<E> next;
    Node<E> prev;
}

// [head] ↔ [Node A] ↔ [Node B] ↔ [Node C] ↔ [tail]

Core API — List + Deque Interface

LinkedList<String> queue = new LinkedList<>();

// ─── LIST OPERATIONS ──────────────────────────────────
queue.add("MSG-001");        // O(1) — adds to tail
queue.add(0, "MSG-000");    // O(1) — adds to head (just re-wire)
queue.get(5);               // O(n) — must traverse 5 nodes
queue.remove(5);            // O(n) — traverse to index 5, then O(1) removal

// ─── DEQUE OPERATIONS (head/tail — all O(1)) ──────────
queue.addFirst("URGENT");  // O(1)
queue.addLast("NORMAL");   // O(1)
queue.peekFirst();          // O(1) — view head without removing
queue.peekLast();           // O(1) — view tail without removing
queue.pollFirst();          // O(1) — remove and return head
queue.pollLast();           // O(1) — remove and return tail
queue.offerFirst("PRIO");  // O(1)
queue.offerLast("STD");    // O(1)

// ─── STACK OPERATIONS ─────────────────────────────────
queue.push("TOP");         // O(1) — same as addFirst
queue.pop();                // O(1) — same as removeFirst
queue.peek();               // O(1) — same as peekFirst

⚠️
Cache Performance Warning

LinkedList nodes are scattered across heap memory — each node allocation is separate. Modern CPUs are optimised for sequential memory access (cache lines). ArrayList elements are contiguous. In practice, ArrayList often outperforms LinkedList even for operations that are theoretically O(1) for LinkedList, because of CPU cache locality. Benchmark before assuming LinkedList is faster.

Production Example — SWIFT Message Priority Queue

/**
 * SWIFT message dispatcher using LinkedList as a Deque.
 * High-priority messages jump to the front, normal ones go to back.
 */
@Service
public class SwiftMessageDispatcher {

    private final LinkedList<SwiftMessage> messageQueue = new LinkedList<>();

    public synchronized void enqueue(SwiftMessage msg) {
        if (msg.getPriority() == Priority.HIGH) {
            messageQueue.addFirst(msg); // O(1) — jump the queue
        } else {
            messageQueue.addLast(msg);  // O(1) — join back of queue
        }
    }

    public synchronized Optional<SwiftMessage> dequeue() {
        return Optional.ofNullable(messageQueue.pollFirst()); // O(1)
    }

    public Optional<SwiftMessage> peek() {
        return Optional.ofNullable(messageQueue.peekFirst()); // O(1)
    }

    public int size() { return messageQueue.size(); }
}

Stack — Last In, First Out

Stacks are fundamental to computing — from function call management to expression parsing to undo systems.

A stack is like a pile of bank statements on your desk. The statement you put on last is the one you pick up first. You can only add to the top (push) or remove from the top (pop). The bottom item is the last one you'll ever reach. This is LIFO: Last In, First Out.

The Two Ways to Implement a Stack in Java

⚠️
Avoid java.util.Stack — Use ArrayDeque Instead

java.util.Stack extends Vector (which is synchronized) and has design issues. The Java documentation itself recommends using ArrayDeque as a stack. ArrayDeque is faster, has a cleaner API, and is not unnecessarily synchronized.

// ✅ Recommended — ArrayDeque as Stack
Deque<String> stack = new ArrayDeque<>();

// PUSH — add to top
stack.push("Validate Payment");      // O(1)
stack.push("Check Funds");           // O(1)
stack.push("Apply FX Rate");         // O(1)

// PEEK — view top without removing
String top = stack.peek();             // O(1) → "Apply FX Rate"

// POP — remove from top
String step = stack.pop();             // O(1) → "Apply FX Rate"

// Other useful operations
boolean empty = stack.isEmpty();       // O(1)
int sz = stack.size();                 // O(1)

// Iterates top-to-bottom (most recently pushed first)
for (String s : stack) System.out.println(s);

Classic Stack Problems

Balanced Parentheses — Common Interview Pattern

/**
 * Validate SWIFT message structure — balanced tags.
 * {1:...}{2:...}{3:...} must all open and close.
 */
public boolean isBalanced(String expression) {
    Deque<Character> stack = new ArrayDeque<>();
    for (char c : expression.toCharArray()) {
        if (c == '{' || c == '[' || c == '(') {
            stack.push(c);
        } else if (c == '}' || c == ']' || c == ')') {
            if (stack.isEmpty()) return false;
            char top = stack.pop();
            if (!matches(top, c)) return false;
        }
    }
    return stack.isEmpty();
}

private boolean matches(char open, char close) {
    return (open == '{' && close == '}')
        || (open == '[' && close == ']')
        || (open == '(' && close == ')');
}

Production Example — Transaction Rollback Stack

/**
 * Undo/rollback mechanism for a payment processing pipeline.
 * Each step is pushed; on failure, steps are popped and reversed.
 */
@Service
public class PaymentPipeline {

    public PaymentResult process(Payment payment) {
        Deque<Runnable> rollbackStack = new ArrayDeque<>();

        try {
            // Step 1
            reserveFunds(payment);
            rollbackStack.push(() -> releaseFunds(payment));

            // Step 2
            convertCurrency(payment);
            rollbackStack.push(() -> reverseConversion(payment));

            // Step 3
            sendSwiftMessage(payment);
            rollbackStack.push(() -> cancelSwiftMessage(payment));

            return PaymentResult.success();

        } catch (Exception e) {
            // Roll back in reverse order — LIFO!
            while (!rollbackStack.isEmpty()) {
                try { rollbackStack.pop().run(); }
                catch (Exception re) { log.error("Rollback failed", re); }
            }
            return PaymentResult.failure(e.getMessage());
        }
    }
}

Queue & Deque — Processing in Order

Queues model real-world waiting lines. Deques extend this with operations on both ends, enabling both stack and queue behaviour.

A Queue is a bank teller queue. Customers join at the back and are served from the front. First come, first served — FIFO: First In, First Out. A Deque (double-ended queue) allows entry and exit at both the front and back — like a VIP bypass lane on both sides of the queue.

Queue Interface & Implementations

// ✅ Best general-purpose Queue/Deque implementation
Queue<String> queue = new ArrayDeque<>();

// ─── QUEUE API ────────────────────────────────────────
// Two flavors: throws exceptions | returns null/false

queue.add("MSG-001");       // throws if capacity exceeded
queue.offer("MSG-002");     // ✅ preferred — returns false if full

queue.remove();              // throws NoSuchElementException if empty
queue.poll();                // ✅ preferred — returns null if empty

queue.element();             // throws if empty
queue.peek();                // ✅ preferred — returns null if empty

// ─── DEQUE API ────────────────────────────────────────
Deque<String> deque = new ArrayDeque<>();

deque.offerFirst("URGENT"); // O(1) — add to front
deque.offerLast("NORMAL");  // O(1) — add to back
deque.pollFirst();           // O(1) — remove from front
deque.pollLast();            // O(1) — remove from back
deque.peekFirst();           // O(1) — view front
deque.peekLast();            // O(1) — view back

ArrayDeque — The Preferred Implementation

ArrayDeque uses a circular buffer (a resizable array with head and tail pointers) internally. This means all head and tail operations are O(1) without the cache-miss problem of LinkedList. It's the go-to implementation for both Queue and Stack use cases.

Production Example — SEPA Payment Processing Queue

/**
 * SEPA payment batch processor using a queue to ensure FIFO
 * processing of payment instructions within a settlement window.
 */
@Service
public class SepaPaymentProcessor {

    private final Queue<SepaPayment> pending = new ArrayDeque<>();
    private final List<SepaPayment> processed = new ArrayList<>();

    public void submitPayment(SepaPayment payment) {
        pending.offer(payment);   // O(1) — join queue
        log.info("Queued: {}", payment.getRef());
    }

    @Scheduled(fixedDelay = 5000)
    public void processBatch() {
        int batchLimit = 100;
        int count = 0;

        SepaPayment payment;
        while (count < batchLimit && (payment = pending.poll()) != null) {
            try {
                settlePayment(payment);
                processed.add(payment);
                count++;
            } catch (Exception e) {
                // re-queue with retry count
                if (payment.getRetries() < 3) {
                    payment.incrementRetries();
                    pending.offer(payment);
                }
            }
        }
        log.info("Batch processed: {} payments", count);
    }
}

HashMap — The Lookup Engine

HashMap gives you O(1) average lookups, inserts, and deletes. Understanding its internals is essential for every Java engineer.

HashMap is like a bank with 16 vault rooms, numbered 0–15. When you open an account, the system hashes your account number to decide which room your file goes in. Later, to find your file, it hashes your number again — jumps to the same room — and retrieves it instantly. No need to search all 16 rooms. The hash function is the magic: same input → always same room. When two accounts hash to the same room, they're stored together in a small list (a bucket).

Internal Mechanics — How HashMap Works

HashMap internally uses an array of buckets. Each bucket is a linked list (or, since Java 8, a red-black tree when a bucket has ≥8 entries). The array starts with 16 slots. When the load factor (default 0.75) is exceeded — i.e., 75% of buckets are occupied — the map rehashes: doubles the array size and redistributes all entries.

// ── Step by step of put("ACCT-001", 5000) ──
// 1. hash = "ACCT-001".hashCode()  →  some integer
// 2. bucket_index = hash & (capacity - 1)   →  e.g., 7
// 3. go to array[7], add the entry to the bucket's list
// 4. if bucket_list.size() >= 8 → convert to red-black tree (Java 8+)
// 5. if (size / capacity) > 0.75 → rehash: double capacity, redistribute

hashCode() and equals() — The Contract

🔴
The Most Important Rule in Java Collections

If two objects are equal (via equals()), they MUST have the same hashCode(). If you override equals(), you MUST override hashCode(). Violating this breaks HashMap, HashSet, and any hash-based collection.

// ❌ BAD — Override equals without hashCode
class AccountKey {
    String id;
    @Override public boolean equals(Object o) { ... } // ✓
    // hashCode() missing! → puts into wrong bucket → can't find entry
}

// ✅ GOOD — Using Lombok or IDE-generated
@EqualsAndHashCode
class AccountKey {
    String id;
    String currency;
}

LinkedHashMap — Ordered HashMap

LinkedHashMap = HashMap + insertion-order doubly linked list. O(1) lookups with predictable iteration — powerful for LRU caches.

LinkedHashMap is a HashMap that also keeps a guest registry. Every time an account is added, its name is written into the registry in order. When you iterate the map, you follow the registry's order — not the random bucket order. The hash lookup is still instant; the registry just records who arrived when.

// Insertion-order (default)
Map<String, Integer> ordered = new LinkedHashMap<>();
ordered.put("EUR", 1);
ordered.put("USD", 2);
ordered.put("GBP", 3);
// Iterates: EUR → USD → GBP (insertion order preserved)

// Access-order — moves most recently accessed to end
Map<String, Integer> accessOrdered = new LinkedHashMap<>(
    16, 0.75f, true  // true = access-order mode
);

// ─── LRU CACHE using LinkedHashMap ────────────────────
int MAX_ENTRIES = 100;
Map<String, AccountBalance> lruCache = new LinkedHashMap<>(
        MAX_ENTRIES, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, AccountBalance> eldest) {
        return size() > MAX_ENTRIES; // evict oldest when full
    }
};

// Usage: identical to HashMap API
lruCache.put("ACC-001", balance1);
AccountBalance b = lruCache.get("ACC-001"); // moves to most-recently-used

TreeMap — Sorted Navigation

TreeMap maintains keys in sorted order using a red-black tree. The tradeoff: O(log n) instead of O(1), but you gain powerful range and navigation operations.

TreeMap is like a bank's sorted ledger book where accounts are listed in alphabetical order. Finding an account by name is O(log n) — you open to the middle, check if you need the left or right half, and repeat. The payoff: you can ask "give me all accounts between 'A' and 'G'" in O(log n + k) — impossible with HashMap.

// Natural key ordering (keys must implement Comparable)
TreeMap<String, BigDecimal> rates = new TreeMap<>();
rates.put("EUR/USD", new BigDecimal("1.0845"));
rates.put("GBP/USD", new BigDecimal("1.2731"));
rates.put("USD/JPY", new BigDecimal("155.20"));
// Iterates in lexicographic order: EUR/USD → GBP/USD → USD/JPY

// Custom ordering — by value date for settlement schedule
TreeMap<LocalDate, List<Payment>> schedule = new TreeMap<>();

// ─── NAVIGATION API — TreeMap's superpower ─────────────
rates.firstKey();                         // O(log n) — smallest key
rates.lastKey();                          // O(log n) — largest key
rates.floorKey("H");                    // O(log n) — greatest key ≤ "H"
rates.ceilingKey("G");                  // O(log n) — smallest key ≥ "G"
rates.lowerKey("G");                    // O(log n) — greatest key strictly < "G"
rates.higherKey("G");                   // O(log n) — smallest key strictly > "G"

// Range views — incredibly powerful for date ranges
SortedMap<String, BigDecimal> sub = rates.subMap("E", "U"); // [E, U)
SortedMap<String, BigDecimal> head = rates.headMap("H");    // everything before "H"
SortedMap<String, BigDecimal> tail = rates.tailMap("H");    // everything from "H" onwards

// Descending view
NavigableMap<String, BigDecimal> desc = rates.descendingMap();

@Service
public class SettlementScheduleService {

    // TreeMap keeps settlement dates sorted automatically
    private final TreeMap<LocalDate, List<Payment>> schedule =
        new TreeMap<>();

    public void schedulePayment(Payment p) {
        schedule.computeIfAbsent(p.getValueDate(), k -> new ArrayList<>())
                .add(p);
    }

    // Get all payments due TODAY and earlier
    public Map<LocalDate, List<Payment>> getDuePayments() {
        return schedule.headMap(LocalDate.now().plusDays(1)); // O(log n)
    }

    // Get next upcoming settlement date
    public Optional<LocalDate> nextSettlementDate() {
        LocalDate tomorrow = LocalDate.now().plusDays(1);
        return Optional.ofNullable(schedule.ceilingKey(tomorrow));
    }
}

  • Navigation methods (floor, ceiling, higher, lower) are its superpower
  • Range views (subMap, headMap, tailMap) are live views — changes reflect in the original
  • Use for: date-keyed schedules, range lookups, ordered event logs, rank-based lookups

Sets — Uniqueness at Scale

Sets are collections that guarantee uniqueness. Three implementations cover the three ordering needs: none, insertion, and sorted.

A Set is like a bank's unique customer registry. You can add a customer, but if they're already registered, nothing changes — no duplicates allowed. HashSet lets you add/check in O(1). TreeSet keeps the registry alphabetically sorted. LinkedHashSet remembers the order customers first registered.

Master heap-based priority ordering for scheduling and top-K problems

PriorityQueue — The Heap

PriorityQueue ensures the highest-priority element is always served first. It's the backbone of scheduling algorithms, Dijkstra's algorithm, and top-K problems.

PriorityQueue is like a hospital emergency room triage system. Patients don't get seen in arrival order — the most critically ill are always moved to the front. No matter how many patients arrive, the current most critical patient surfaces in O(log n). Internally this is a binary heap — a special tree where the root is always the minimum (or maximum) element.

Heap Internals

A min-heap (Java default) stores elements so that the smallest element is always at position 0. The heap is stored as an array, with the parent of element at index i at index (i-1)/2, and children at 2i+1 and 2i+2. After each add or remove, the heap sifts elements up or down to maintain the heap property.

// ─── MIN-HEAP (default — smallest element at top) ─────
PriorityQueue<Integer> minHeap = new PriorityQueue<>();
minHeap.offer(500); minHeap.offer(100); minHeap.offer(800); minHeap.offer(300);
minHeap.peek();  // O(1) → 100 (minimum)
minHeap.poll();  // O(log n) → removes 100, heap restructures
minHeap.peek();  // O(1) → 300 (new minimum)

// ─── MAX-HEAP (reverse order) ─────────────────────────
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());
maxHeap.offer(500); maxHeap.offer(100); maxHeap.offer(800);
maxHeap.peek();  // O(1) → 800 (maximum)

// ─── CUSTOM PRIORITY — by payment amount ──────────────
PriorityQueue<Payment> byAmount = new PriorityQueue<>(
    Comparator.comparing(Payment::getAmount).reversed() // highest first
);

// ─── FULL API ─────────────────────────────────────────
PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.offer(5);        // O(log n) — add
pq.peek();          // O(1) — view head without removing
pq.poll();          // O(log n) — remove and return head
pq.contains(5);    // O(n) — linear search!
pq.remove(5);      // O(n) — remove specific element
pq.size();          // O(1)
pq.isEmpty();       // O(1)
⚠️
PriorityQueue does NOT sort on iteration

Iterating a PriorityQueue with a for-each loop does NOT give you elements in priority order. Only poll() removes them in priority order. To iterate in order: while(!pq.isEmpty()) { System.out.println(pq.poll()); }

Classic Problem — Top K Largest Payments

/**
 * Find top K largest payments efficiently.
 * Approach: min-heap of size K — maintain K largest seen so far.
 * Time: O(n log K)  Space: O(K)
 */
public List<Payment> topKPayments(List<Payment> payments, int k) {
    // Min-heap — smallest of the "top K" sits at root
    PriorityQueue<Payment> heap = new PriorityQueue<>(
        Comparator.comparing(Payment::getAmount)
    );

    for (Payment p : payments) {
        heap.offer(p);
        if (heap.size() > k) heap.poll(); // evict smallest — O(log K)
    }

    // Collect remaining K elements (in reverse to get descending order)
    List<Payment> result = new ArrayList<>(heap);
    result.sort(Comparator.comparing(Payment::getAmount).reversed());
    return result;
}

// Why not sort()? O(n log n) vs O(n log K) — huge win when K << n

The Collections Utility Class

java.util.Collections is a toolbox of static helper methods for sorting, searching, shuffling, and wrapping collections. Know every major method.

Collections.fill(amounts, 0);                        // O(n) — all elements to value

List<String> empty  = Collections.emptyList();                // immutable, 0 elements
Set<String> emptySet = Collections.emptySet();
  • Collections.sort() uses TimSort — O(n log n), stable, excellent for nearly-sorted data
  • Collections.binarySearch() requires a sorted list — undefined behavior on unsorted lists
  • Unmodifiable wrappers are shallow — elements themselves can still be mutated if they're mutable objects
  • For production thread safety, prefer ConcurrentHashMap over Collections.synchronizedMap()
  • Collections.emptyList() is better than new ArrayList<>() for return values that are never populated

Interview Master Guide

The most important concepts, patterns, and questions asked in Java engineering interviews — from junior to senior level.

Tier 1 — Core Conceptual Questions

Frequently Asked — Every Level
What is the difference between HashMap and Hashtable?
HashMap is not synchronised (not thread-safe), allows one null key and multiple null values, and is faster. Hashtable is synchronised (thread-safe via method-level locking), does not allow null keys or values, and is slower. In modern Java, you should use ConcurrentHashMap instead of Hashtable for thread-safe maps — it uses segment-level locking for much better concurrency.
Frequently Asked — Every Level
What happens when two keys have the same hashCode in a HashMap?
This is a hash collision. Both entries go into the same bucket (array slot). Before Java 8, collisions are stored as a linked list — O(n) worst-case lookup for that bucket. Since Java 8, when a bucket's linked list reaches 8 entries, it's converted to a red-black tree, reducing worst-case lookup to O(log n). The equals() method is then used to find the exact entry within the bucket.
Must Know
Why must you override hashCode() when you override equals()?
The hashCode contract states: if two objects are equal (via equals()), they MUST return the same hashCode. HashMap uses hashCode to determine the bucket, then equals() to find the exact entry within the bucket. If two equal objects have different hashCodes, they land in different buckets — the map will never find the second entry when looking up the first, breaking the map's correctness completely.
Senior Level
What is the time complexity of HashMap operations, and when can they degrade?
Average case is O(1) for get/put/remove. This degrades when: (1) many keys collide into the same bucket — worst case O(n) before Java 8, O(log n) after with tree conversion. (2) The load factor threshold is exceeded, triggering rehashing — O(n) for that operation. To prevent degradation: use well-distributed hashCode implementations, choose appropriate initial capacity (new HashMap<>(expectedSize / 0.75 + 1)), and avoid mutable objects as keys.
Frequently Asked
When would you choose LinkedList over ArrayList?
Rarely in practice. Theoretically, LinkedList is better for O(1) insertions/deletions at the head or tail. However, ArrayList's cache locality means it often outperforms LinkedList even in those scenarios. Choose LinkedList only when: you need a Deque (double-ended queue), you have a confirmed performance bottleneck from ArrayList's head insertions, or you're implementing a specific algorithm requiring efficient node splicing. Otherwise, ArrayList wins.
Must Know
What is ConcurrentModificationException and how do you avoid it?
CME is thrown when you structurally modify a collection (add/remove elements) while iterating over it with a for-each loop or iterator, except through the iterator's own remove() method. The collection tracks a modCount. The iterator checks it on each next() call — if mismatched, it throws CME. Solutions: use Iterator.remove() for safe removal, use removeIf() (Java 8+), collect items to remove then call removeAll(), or use a thread-safe collection like CopyOnWriteArrayList.
Must Know
What is the difference between fail-fast and fail-safe iterators?
Fail-fast iterators (ArrayList, HashMap, HashSet): immediately throw ConcurrentModificationException if the collection is structurally modified during iteration. They operate directly on the collection. Fail-safe iterators (ConcurrentHashMap, CopyOnWriteArrayList): iterate on a snapshot or allow concurrent modification without exceptions. They may not reflect changes made after iteration started. The tradeoff: fail-safe is safer in concurrent scenarios but uses more memory and may be stale.

Tier 2 — Coding Problems

Common Coding Interview
Two Sum — Find two numbers that add to a target
Use a HashMap for O(n) solution: for each number, check if (target - number) exists in the map. If yes, found the pair. If no, add the number to the map. This avoids the brute-force O(n²) nested loop.
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (seen.containsKey(complement)) {
            return new int[]{seen.get(complement), i};
        }
        seen.put(nums[i], i);
    }
    return new int[]{};
} // O(n) time, O(n) space
Common Coding Interview
Group Anagrams
Sort each word to create a canonical key. Group words by their sorted form using a HashMap<String, List<String>>.
public List<List<String>> groupAnagrams(String[] strs) {
    Map<String, List<String>> map = new HashMap<>();
    for (String s : strs) {
        char[] chars = s.toCharArray();
        Arrays.sort(chars);
        String key = new String(chars);
        map.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
    }
    return new ArrayList<>(map.values());
} // O(n × k log k) where k = max word length
Common Coding Interview
LRU Cache Implementation
Use LinkedHashMap in access-order mode. Override removeEldestEntry() to evict the least-recently-used entry when the cache is full. All operations are O(1).
class LRUCache {
    private final Map<Integer, Integer> cache;

    public LRUCache(int capacity) {
        cache = new LinkedHashMap<>(capacity, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<Integer, Integer> eldest) {
                return size() > capacity;
            }
        };
    }

    public int get(int key) {
        return cache.getOrDefault(key, -1); // O(1) — moves to most-recent
    }

    public void put(int key, int value) {
        cache.put(key, value); // O(1) — auto-evicts if over capacity
    }
}

Tier 3 — Advanced Design Questions

Senior/Lead Level
How would you design a high-throughput payment deduplication system handling 100K messages/second?
Single JVM: ConcurrentHashMap with TTL-based eviction using scheduled cleanup. The map stores endToEndId → timestamp. Cleanup removes entries older than the deduplication window (e.g., 24h).

Distributed: Redis with SET NX (set-if-not-exists) and TTL. SETNX is atomic — a message ID is either claimed or rejected. For billions of IDs, consider a Bloom Filter (probabilistic, space-efficient, no false negatives) as a first pass before querying the persistent store.

Key data structure choice: ConcurrentHashMap (JVM-local), Redis SET (distributed), Bloom Filter (ultra-high-scale with acceptable false-positive rate).
Senior Level
Explain how Java 8 improved HashMap worst-case performance.
Before Java 8, colliding entries in a bucket were stored as a linked list — O(n) worst-case lookup. A hash DOS attack could degrade all HashMap operations to O(n) by crafting inputs with the same hash. Java 8 introduced tree bins: when a bucket's linked list grows to 8 entries, it converts to a red-black tree — reducing worst-case lookup to O(log n). When entries drop below 6, it reverts to a linked list. Additionally, Java 8 improved the hash function to better distribute String keys.

Quick-Fire Question Bank

Master Quiz — All Chapters

10 Questions

Q1. Which collection would you use to count the frequency of each word in a text file?

Reveal Answer
HashMap<String, Integer>. Use merge(word, 1, Integer::sum) or compute to increment counts. For sorted output by frequency, collect into a list and sort, or use TreeMap if you want word-alphabetical order.

Q2. What is the output order when iterating a HashMap?

Reveal Answer
Undefined — there is no guaranteed iteration order in HashMap. The order depends on hash codes and internal bucket distribution. Never rely on HashMap iteration order. Use LinkedHashMap for insertion order or TreeMap for sorted order.

Q3. What is the initial default capacity and load factor of HashMap?

Reveal Answer
Initial capacity = 16, load factor = 0.75. This means the map rehashes when 12 entries (16 × 0.75) are added. Always use a power of 2 for initial capacity — HashMap requires this for its bitwise bucket indexing.

Q4. Can a HashSet contain null? What about TreeSet?

Reveal Answer
HashSet: yes, one null element allowed. TreeSet: no — null would cause a NullPointerException when compared using compareTo() during tree navigation. Exception: if you provide a custom Comparator that handles null explicitly.

Q5. What is the difference between poll() and remove() on a Queue?

Reveal Answer
Both remove the head element. remove() throws NoSuchElementException if the queue is empty. poll() returns null if empty. Always prefer poll() — defensive programming avoids unexpected exceptions.

Q6. Why is ArrayList not thread-safe and how do you fix it?

Reveal Answer
ArrayList is not synchronized — concurrent modifications can corrupt internal state (wrong size count, missed elements). Fix options: (1) CopyOnWriteArrayList — thread-safe, creates a new copy on each write, ideal for read-heavy scenarios. (2) Collections.synchronizedList() — wraps with coarse lock, must manually synchronize iteration. (3) External synchronization with synchronized blocks around all access.

Q7. What data structure would you use to find the median of a stream of numbers efficiently?

Reveal Answer
Two heaps: a max-heap for the lower half and a min-heap for the upper half. Maintain size balance (differ by at most 1). Median is the top of the larger heap, or average of both tops if equal size. O(log n) per insertion, O(1) median query.

Q8. Explain the difference between Comparable and Comparator in Java collections.

Reveal Answer
Comparable is implemented by the class itself (natural ordering) via compareTo(). Comparator is an external comparison strategy passed to sorted collections or sort methods. Use Comparable when there's one obvious natural order. Use Comparator for multiple sort orders, sorting third-party classes you can't modify, or when the natural order doesn't suit your need (e.g., TreeMap with custom Date comparison).

Q9. What is the space complexity of a HashMap with n entries?

Reveal Answer
O(n) for the entries themselves, but with significant overhead. The backing array is always larger than n due to the load factor (at most 75% full). Each entry is a Node object (key + value + hash + next pointer). Total memory ≈ n × (entry object overhead) + array capacity × reference size. In practice, HashMaps use significantly more memory per element than arrays or ArrayLists.

Q10. How would you efficiently find all accounts with a balance between £5,000 and £50,000?

Reveal Answer
If accounts are stored by balance as the key: TreeMap<BigDecimal, Account> — use subMap(5000, 50001) for O(log n + k) where k is the result size. If accounts are stored by ID: HashMap<String, Account> — you must scan all values, which is O(n). The key lesson: if range queries on a specific field are frequent, store or index by that field in a TreeMap.
